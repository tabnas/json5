# Agents Guide — json5

## What this project is

`@tabnas/json5` is a **JSON5 grammar plugin** for the
[`tabnas`](https://github.com/tabnas/parser) parsing engine. It accepts
[JSON5](https://json5.org) syntax — comments (`//`, `/* */`), trailing
commas, single-quoted strings, unquoted ECMAScript IdentifierName keys,
hex numbers, leading/trailing decimal points, `+`/`-` signs,
`Infinity` / `NaN`, and string line continuations — over and above
standard JSON.

It is **not** standalone: it layers on the
[`@tabnas/jsonic`](https://github.com/tabnas/jsonic) base grammar (the
relaxed-JSON `val`/`map`/`list`/`pair`/`elem` rules) and the
[`@tabnas/parser`](https://github.com/tabnas/parser) engine, then
**tightens** jsonic toward the JSON5 spec: it drops jsonic's implicit
top-level `a:1` / `1,2` forms (`rule.exclude: 'imp'`), restricts the
`VAL`/`KEY` token sets, and rejects unquoted text that is not a valid
IdentifierName. The default option set is a strict JSON5 configuration;
several relaxations (hash comments, backtick strings, octal/binary/
separator numbers) are off by default and opt-in.

There are two implementations that must behave identically —
TypeScript (canonical) and a Go port.

## The tabnas engine dependency

Both runtimes depend on the unpublished `@tabnas` siblings via a
**sibling checkout** (the standard tabnas dev model until the packages
publish tagged releases):

- TypeScript: `@tabnas/parser` and `@tabnas/jsonic` are declared as
  `peerDependencies` (`">=2"`) in `ts/package.json` and mirrored as
  `file:../../parser/ts` / `file:../../jsonic/ts` devDependencies for
  local builds (npm ≥7 / Node ≥24 auto-installs peers; `engines.node`
  is `">=24"`). `@tabnas/debug` and `@tabnas/railroad` are dev-only
  `file:` devDependencies (railroad to regenerate
  `ts/doc/grammar.{svg,txt}`; debug is present but **not** wired into a
  `debug.model()` test in this repo — see below).
- Go: `go/go.mod` requires `github.com/tabnas/jsonic/go` with
  `replace github.com/tabnas/jsonic/go => ../../jsonic/go`. That is the
  module's **only** tabnas dependency — the Go jsonic package re-exports
  the engine types (`jsonic.Jsonic`, `jsonic.AltSpec`, `jsonic.Tin`, …),
  so `json5.go` imports `jsonic`, not `parser`, directly.

Clone the siblings (`parser`, `jsonic`, plus `debug`/`railroad`) next to
this repo and build their TS first. CI does this for you (see below).

## Authority and alignment rules

**TypeScript is canonical. Go is a port of it.** When you change
behaviour:

1. Change `ts/json5-grammar.jsonic` (for grammar changes — see the embed
   section) or `ts/src/json5.ts` (for code-level overrides) first.
2. Port the same change to `go/json5.go`.
3. Add/extend a fixture in `test/json5-tests/` so both runtimes assert
   the new behaviour. The vendored corpus is the parity contract; both
   suites walk it (TS: [`ts/test/suite.test.ts`](ts/test/suite.test.ts)
   at `../../test/json5-tests`; Go:
   [`go/suite_test.go`](go/suite_test.go) `TestOfficialSuite` at
   `../test/json5-tests`).
4. Mirror any new unit cases across `ts/test/json5.test.ts` and
   `go/json5_test.go`.
5. Run both suites and confirm green.

Do not let the Go behaviour drift from TS.

### The conformance corpus (`test/json5-tests/`)

The fixture extension encodes the expected outcome (see
`test/json5-tests/README.md`):

- `.json` — valid JSON, must parse.
- `.json5` — valid JSON5, must parse.
- `.js` — valid ES5 but explicitly disallowed by JSON5, must error.
- `.txt` — invalid everywhere, must error.

Both suites walk the tree and assert parse-success vs parse-error per
extension; they **skip** gracefully if the corpus directory is absent.

## The grammar is embedded — never hand-edit the embedded block

`ts/json5-grammar.jsonic` is embedded verbatim into **both**
`ts/src/json5.ts` and `go/json5.go`, between these markers:

```
// --- BEGIN EMBEDDED json5-grammar.jsonic ---
...
// --- END EMBEDDED json5-grammar.jsonic ---
```

Edit `ts/json5-grammar.jsonic`, then run the embed step. Never edit the
text between the markers by hand — it will be overwritten.

```bash
cd ts && node embed-grammar.js   # writes into ts/src/json5.ts AND go/json5.go
```

`npm run build` runs the embed step first (`node embed-grammar.js && tsc
--build src test`), so a normal TS build keeps both files in sync. The
TS side escapes the grammar into a template literal; the Go side splits
on backticks and re-joins (`+ "`" +`) because a Go raw string cannot
contain a backtick.

### Placeholders are patched in code

The grammar file uses bare UPPER_SNAKE_CASE placeholder strings
(`JSON5_WHITESPACE`, `JSON5_LINE_TERMINATOR`, `JSON5_ROW_CHARS`,
`JSON5_QUOTE_CHARS`, `JSON5_MULTI_QUOTE_CHARS`) for the whitespace and
quote character sets, because those sets contain code points the
jsonic grammar parser cannot round-trip losslessly. The plugin
substitutes the real character constants after parsing the grammar text
and before applying it (`grammarDef.options.space.chars = JSON5_WHITESPACE`,
etc.). `@`-prefixed names (`@fixed-check`, `@text-check`,
`@parse-trailing-dec-exp`, `@parse-uppercase-hex`, and the `@/regex/`
forms) resolve against the `ref` map.

## Repo-specific gotchas

### String line continuations are stripped at the source level (NOT via the escape map)

A JSON5 string line continuation — a backslash immediately followed by a
LineTerminatorSequence (`CRLF`, `CR`, `LF`, `U+2028` LS, `U+2029` PS) —
produces an empty string so the string spans lines. This **cannot** be
done through the escape map: the lexer drops any escape entry whose
replacement is the empty string. So both runtimes strip continuations
from the source before lexing:

- **TS** (`ts/src/json5.ts`, `fixedCheck`, wired as `fixed.check`): a
  one-time per-parse rewrite, `src.replace(...)`. The regex is built with
  the **string form** `new RegExp('\\\\(?:\\r\\n|[\\r\\n\\u2028\\u2029])', 'g')`,
  **not** a regex literal — a literal `U+2028`/`U+2029` in JS source is a
  line break and breaks parsing (historically; even where tolerated now,
  keep the string form for safety and to mirror the explicit-escape
  intent). Do not "simplify" this into a `/.../u` literal.
- **Go** (`go/json5.go`): a package-level `json5LineContinuations =
  strings.NewReplacer(...)` with five backslash-prefixed pairs (`\CRLF`,
  `\CR`, `\LF`, `\U+2028`, `\U+2029`, each mapped to `""`). **CRLF is
  listed first** so the two-character sequence is matched before a lone
  `CR`/`LF`. Keep that ordering.

### Number shapes the built-in lexer misses are matched by regex value-defs

Trailing-decimal-with-exponent (`5.e4`) and uppercase `0X` hex are not
recognised by the engine's number lexer, so the grammar registers
regex-matched `value.def` entries (`trailingDecExp` → `parseTrailingDecExp`,
`uppercaseHex` → `parseUppercaseHex`) to give both runtimes identical
behaviour on those forms.

### `Infinity` / `NaN` keywords are layered on in code, not in the grammar file

The `Infinity` / `+Infinity` / `-Infinity` / `NaN` / `+NaN` / `-NaN`
value defs hold actual JS numbers (`Infinity`, `-Infinity`, `NaN`), which
cannot round-trip through the jsonic grammar parser, so they are added in
code under the `infinity` option (default on).

### Code-level rule trims the grammar can't express

- `pair` loses its leading-comma / `jsonic` open alt (so `{,}` fails) and
  gains an after-open validator that rejects `#TX` keys whose source is
  not a valid JSON5 IdentifierName (`isValidIdentifierName`).
- When `requireValue` is set (default), `val` loses its `#ZZ jsonic` alt
  and the parser's `start` is wrapped so an empty/whitespace-only source
  errors with code `json5_empty` / `json5_no_value`.
- `textCheck` (wired as `text.check`) halts lexing on unquoted text that
  is neither a valid IdentifierName start nor a registered value keyword/
  regex, raising "unexpected character".
- Backtick is pruned from the quote sets unless `backtickString` is on
  (jsonic's option merge is additive for char sets, so it must be removed
  explicitly).

## Build & test

TypeScript (from `ts/`):

```bash
npm install            # auto-installs peers; resolves file: siblings
npm run build          # embeds grammar, then tsc --build src test
npm test               # node --test over dist-test/*.test.js
```

Go (from `go/`):

```bash
go build ./...
go test -v ./...       # unit tests + the vendored json5-tests corpus
```

The repo root [`Makefile`](Makefile) wraps both: `make build|test` run
the TS and Go halves; `make publish-go V=x.y.z` injects `V` into the
`const Version` in `go/json5.go` (currently `0.1.1`) and tags `go/vX.Y.Z`;
`make tags-go` lists Go tags. The TS package version is `0.1.0`
(`ts/package.json`).

## Tests in this repo

- `ts/test/json5.test.ts` / `go/json5_test.go` — unit suites (mirror each
  other).
- `ts/test/suite.test.ts` / `go/suite_test.go` — the vendored JSON5
  conformance corpus runners.
- `ts/test/doc-examples.test.ts` — extracts fenced `js`/`javascript`
  blocks containing `// =>` assertions from `README.md`, `ts/README.md`,
  `go/README.md`, and `ts/doc/*.md`, and runs them; keep doc examples
  correct.

Note: unlike most tabnas grammar repos, this repo has **no**
`debug-model.test.ts` — `@tabnas/debug` is a dev dependency (pulled in by
the shared dev setup) but is not exercised by a structured-model test
here.

## CI

`.github/workflows/build.yml` has two jobs, neither publishing to npm:

- **build** (Ubuntu/Windows/macOS, Node 24): sets
  `git config --global core.autocrlf false` (CRLF corrupts the corpus
  fixtures), git-clones the tabnas closure
  (`parser debug json abnf railroad jsonic`) as siblings, `npm i && npm
  run build --if-present` each in topo order, then `npm test` in
  `json5/ts`.
- **build-go** (Ubuntu/macOS, Go 1.24): clones the same siblings,
  mirrors `admin/scripts/link.sh` by creating `vendor/` symlinks for any
  `../vendor/` replaces and a `go work` over every non-vendor-replaced
  module, then `go build ./...` / `go test -v ./...` in `json5/go`.
